# Architecture

A concise map of the codebase for anyone (including a future BUILD/QA run)
picking this up cold.

## Shape

```
src/            politebot-core: the Rust engine, compiled to wasm
  lib.rs        wasm-bindgen exports — the only surface JS talks to
  analyzer.rs   static analysis of pasted scraper source (Verdict/Finding)
  robots.rs     from-scratch robots.txt parser (RobotsRules)
  fixer.rs      one-click "add User-Agent" fix-diff generator
tests/          integration tests, one file per module + a fuzz-style
                panic-freedom regression suite
site/           the static front end — plain HTML/CSS/JS, no build step
  index.html    page structure: code pane, verdict pane, robots-check form
  app.js        wires the wasm module to the DOM
  styles.css    terminal-mono design tokens + component styles
  pkg/          generated by scripts/build-wasm.sh — gitignored, not source
scripts/
  build-wasm.sh cargo build --release --target wasm32-unknown-unknown,
                then wasm-bindgen into site/pkg/
docs/           VISION, BACKLOG, DESIGN, this file
```

## Data flow

1. User pastes source into `#source-input` (`site/index.html`).
2. `app.js` debounces 150ms, then calls the wasm export `score_scraper(source)`
   (`src/lib.rs` → `analyzer::analyze`), which returns a JSON string
   `{"verdict": "red"|"yellow"|"green", "findings": [{"line", "message"}]}`.
3. `renderScore` in `app.js` paints the verdict banner, the findings list
   (via the `<template id="finding-template">`), and the line-gutter markers
   (`renderGutter`) — all client-side, no network round trip.
4. Each finding is also passed to the wasm export `suggest_fix(source, line)`
   (`src/lib.rs` → `fixer::suggest_user_agent_fix`); when it returns a
   non-empty JSON payload, the finding gets a "fix" button that reveals a
   diff and a "copy fixed code" clipboard action.
5. Separately, the robots.txt cross-check form calls `check_robots(text,
   agent, path)` (`src/lib.rs` → `robots::parse` + `RobotsRules::is_allowed`)
   live as the user types a target path.

Nothing pasted ever leaves the browser — there is no backend and no network
call anywhere in this flow.

## Core engine (`src/`)

- **`analyzer.rs`** — `analyze(source) -> PolitenessScore`. Line-scans for
  request-call signals (`REQUEST_SIGNALS`), User-Agent signals
  (`USER_AGENT_SIGNALS`), default/library User-Agent strings
  (`DEFAULT_USER_AGENT_SIGNALS`), and rate-limit/backoff signals
  (`BACKOFF_SIGNALS`, now including `tenacity`/`@retry`). Source with no
  recognized request pattern short-circuits to a single Yellow finding
  instead of guessing Red. Verdict is derived from `findings.len()`
  (0 → Green, 1 → Yellow, 2+ → Red).
- **`robots.rs`** — `parse(text) -> RobotsRules`. Groups `User-agent`
  blocks and answers `is_allowed`/`crawl_delay` by longest-prefix
  `Allow`/`Disallow` precedence, falling back to the `*` group, falling back
  to allow-all if nothing parses (empty/malformed input is never an error).
- **`fixer.rs`** — `suggest_user_agent_fix(source, line) -> Option<FixSuggestion>`.
  Patches Python `requests.get/post(...)` calls (adds a `headers` dict above
  the call) and JS `fetch(url)` calls (inlines a headers options object),
  bracket-depth-aware so nested calls patch correctly. Returns `None` for
  call shapes it doesn't recognize rather than guessing at a broken patch.
- **`lib.rs`** — the wasm-bindgen boundary. Every export takes/returns
  primitives (`&str`, `String`, `bool`) and hand-rolls JSON (see
  `to_json`/`json_string` in each module) so no serde dependency is needed
  across the boundary.

## Front end (`site/`)

Static, dependency-free, relative-path-only (deployable to a subpath like
`apps.charliekrug.com/polite-bot/`). `app.js` is the only script; it imports
the generated `pkg/politebot_core.js` ES module, initializes the wasm
instance, and wires:

- the paste editor + its line-number gutter (`renderGutter`, scroll-synced),
- the verdict banner + findings list + per-finding fix panel,
- the "try an example" buttons (`EXAMPLES` in `app.js`),
- the robots.txt cross-check form.

Visual direction and tokens live in `docs/DESIGN.md`; `styles.css` is the
only stylesheet and defines all of them as CSS custom properties on `:root`.

## Running it

```sh
cargo test                 # Rust unit + integration tests
cargo fmt --check           # formatting
cargo clippy --all-targets  # lints
scripts/build-wasm.sh       # builds site/pkg/ (gitignored build output)
# then open site/index.html in a browser
```
